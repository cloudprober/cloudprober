From zero to a probe in 30 seconds

Drop a config file. Run a container. Scrape metrics.

1 · cloudprober.cfg

# Probe two hosts every 10s and alert on Slack.
probe {
  name: "homepage"
  type: HTTP
  targets {
    host_names: "cloudprober.org,github.com"
  }
  interval_msec: 10000

  alert {
    name: "homepage-down"
    notify { slack { webhook_url: "https://hooks.slack.com/..." } }
  }
}

surfacer { type: PROMETHEUS }

2 · run it

$ docker run -p 9313:9313 \
    -v $(pwd)/cloudprober.cfg:/cfg/cloudprober.cfg \
    cloudprober/cloudprober

3 · scrape /metrics

$ curl localhost:9313/metrics
total{probe="homepage",dst="cloudprober.org"} 24
success{probe="homepage",dst="cloudprober.org"} 24
latency{probe="homepage",dst="cloudprober.org"} 41.2

Trusted by leading organizations

⚡️ Fast native checks

Cloudprober comes with efficient, highly scalable, built-in probes for common types of checks: HTTP/S, gRPC, DNS, PING, TCP, UDP.

🧩 Flexible custom probes and extensibility

Run any binary or script as a probe (see external probe). You can also add your own probe types and target types based on your organization's needs.

🔌 Easy integration

Out of the box, config based, integration with many popular monitoring and alerting systems: Prometheus, Grafana, DataDog, PostgreSQL, AWS Cloudwatch, Stackdriver, PagerDuty, Opsgenie, and more.

🎯 Automated targets discovery

Reliable, automated targets discovery for Kubernetes and GCP resources. Or, use file targets to define probes once and have probes be automatically updated when targets file changes.

👥 Open Source

Cloudprober is open-source and free to use. It is actively maintained, and has a growing community of users and contributors.

🚀 Deployment friendly

Written entirely in Go. Available as a standalone static binary, through a container image, or Helm charts. Multiple ways to configure, including environment variables.
